\chapter*{Abstract}
\addcontentsline{toc}{chapter}{Abstract}

Online learning has become a primary mode of self-directed education for university students, yet the sheer volume of available resources often makes it difficult to know what to study, in what order, and which materials are actually worth the time. This thesis presents \textbf{Opic}, a topic-structured learning platform with user-driven content personalisation, that addresses these three challenges through structured topic roadmaps, a community-driven content pipeline with administrator review, and a collaborative filtering recommendation engine.

The system is built on a client-server architecture: a React single-page application communicates with a FastAPI backend over a JSON REST API, persisting data to a PostgreSQL relational database. The recommendation engine implements user-based collaborative filtering (UBCF) with cosine similarity, augmented by a Bayesian average popularity score to handle sparsity, and a cold-start fallback driven by an onboarding quiz taken by new users. When a user marks a topic as complete, a knowledge quiz generated by the Google Gemini API is presented; users who answer three or more questions incorrectly are prompted to review the material before proceeding.

The platform was deployed to free-tier cloud infrastructure (Render for the backend, Vercel for the frontend) and tested end-to-end with multiple user accounts. All fifteen functional requirements were successfully implemented and verified on the live deployment. The report documents the system architecture, database design, use case model, implementation details of the core algorithms, and a frank discussion of the challenges and limitations encountered during development.

\textbf{Keywords:} recommender systems, collaborative filtering, learning platform, FastAPI, React, cold-start problem, Bayesian average, knowledge quiz

\chapter{Introduction}

\section{Context and Motivation}

The amount of learning content available online today is almost overwhelming. Platforms like YouTube, Coursera, and Medium publish thousands of articles, videos, and tutorials every day \cite{coursera2023}. For a student trying to learn something new -- say, data structures or web development -- the real problem is not finding content; it is figuring out \textit{what to learn, in what order, and what is actually worth their time}.

The challenge of information overload in online education has been well documented. Drachsler and Greller \cite{drachsler2016} described this as the ``paradox of choice'' in technology-enhanced learning: the abundance of resources does not guarantee better learning outcomes and can in fact increase cognitive load and learner disorientation. Burke \cite{burke2002} similarly identified the inability to surface personally relevant content as one of the central unsolved problems in web-based learning environments.

Recommender systems have been proposed as a solution to this problem since the mid-1990s, beginning with the seminal GroupLens project \cite{resnick1994}, which applied collaborative filtering to Usenet news articles. Since then, collaborative filtering has become the dominant paradigm for personalised recommendation \cite{koren2009}, deployed at scale by commercial platforms such as Netflix, Spotify, and Amazon. Its application to educational content, however, remains less mature, with most academic platforms still relying on manually curated content hierarchies \cite{tang2015}.

This is the problem that motivated the development of \textbf{Opic}. The name comes from the word \textit{topic} -- the fundamental unit of learning in the system. Opic is a web-based learning platform that organises knowledge into structured topic roadmaps, lets the community contribute and rate resources, and uses a recommendation engine to surface relevant materials for each individual user based on their behaviour and interests.

The system was built with university students in mind, particularly those studying Information and Communication Technology (ICT), Data Science (DS), and Business Information Technology (BIT), where self-directed learning is common and the volume of online material can be disorienting. Self-directed learning in higher education has been extensively studied \cite{zimmerman2002}, with evidence suggesting that learners benefit significantly from systems that scaffold their path through a subject domain \cite{brusilovsky2007}.

\section{Objectives}

The primary goal of this project was to design and build a functional, deployable learning platform that addresses three core problems a student faces when navigating online learning materials:

\begin{enumerate}
  \item \textbf{Navigation:} Not knowing what to learn first or next.
  \item \textbf{Quality filtering:} Not knowing which resources are actually good.
  \item \textbf{Personalisation:} Receiving the same generic content regardless of background or interests.
\end{enumerate}

These three challenges are addressed respectively through structured topic roadmaps, a community-driven rating and admin-review pipeline, and a user-based collaborative filtering (UBCF) recommendation engine. The UBCF approach was chosen over content-based filtering because it can leverage the collective judgement of many users rather than requiring expert content tagging, and over matrix factorisation because it is interpretable and does not require offline training \cite{herlocker2004}. The cold-start problem -- the inability to recommend to new users with no history -- is handled through an onboarding quiz that seeds the system with topic-level preference signals \cite{schein2002}.

\section{Expected Outcome}

The following outcomes were achieved by the end of the project:

\begin{itemize}
  \item A fully deployed, publicly accessible web application where any user can register, complete the onboarding quiz, browse structured topic roadmaps, and access curated resources.
  \item A working recommendation engine that provides personalised resource suggestions from the first login (via onboarding cold-start) and improves as users rate and complete more content.
  \item A complete admin pipeline where content can be submitted by any registered user, reviewed by admins, and published or rejected -- maintaining content quality without manual curation of the entire library.
  \item A progress tracking system that allows users to mark topics as completed and view their learning activity over a 30-day period.
  \item A JWT-based authentication system with a secure password reset flow \cite{ietfjwt}, protecting all state-mutating endpoints behind role-aware access control. Passwords are stored as bcrypt hashes following OWASP recommendations \cite{owasp}.
  \item A clean client-server deployment on free-tier cloud infrastructure (Vercel for the frontend, Render for the backend and database), with CI/CD triggered automatically on every push to the main branch.
\end{itemize}

\section{Desired Features and Design Decisions}

At the start of the project, the following features were identified as essential requirements. Each feature directly informed an architectural and design decision that is elaborated in Chapters 4 and 5.

\begin{itemize}
  \item \textbf{AI-generated knowledge quiz} -- drove the design of the \texttt{GET /topics/\{id\}/quiz} endpoint, the Gemini API integration \cite{gemini2024}, and the \texttt{QuizModal} component on the topic detail page. The decision to gate topic completion behind a quiz was motivated by research showing that retrieval practice (testing oneself on material) substantially improves retention compared to passive re-reading \cite{roediger2006}.
  \item \textbf{User registration, login, and JWT-based authentication} -- drove the design of the \texttt{users} table, the \texttt{/auth} API module, and the React Context-based auth state management on the frontend.
  \item \textbf{Structured topic roadmaps with prerequisite relationships} -- drove the design of the \texttt{topics} and \texttt{topic\_prerequisites} tables and the topic detail page layout. Prerequisite sequencing is a well-established practice in intelligent tutoring systems \cite{brusilovsky2007}.
  \item \textbf{Community resource submission} -- drove the resource submission form and the \texttt{resources} table with a \texttt{status} field to gate visibility. Crowdsourced content curation has been shown to be effective at surfacing high-quality materials at scale \cite{anderson2012}.
  \item \textbf{Admin review workflow} -- drove the dedicated admin review page and the \texttt{POST /resources/\{id\}/review} endpoint with role checking.
  \item \textbf{Star ratings and engagement tracking} -- drove the \texttt{ratings} and \texttt{engagements} tables and the engagement-score formula used in the recommender. Combining explicit ratings with implicit signals such as time spent and completion rate is a standard technique for improving recommender accuracy \cite{hu2008}.
  \item \textbf{Personalised recommendations using collaborative filtering} -- drove the design of the recommender service, including the user-resource matrix, cosine similarity computation \cite{salton1988}, and Bayesian average popularity score to handle the cold-resource problem \cite{imdb_bayes}.
  \item \textbf{Cold-start onboarding quiz} -- drove the \texttt{onboarding\_questions} and \texttt{onboarding\_answers} tables and the topic-boosting logic in the cold-start fallback. The cold-start problem is one of the most studied open challenges in recommender systems research \cite{schein2002, lam2008}.
  \item \textbf{Progress tracking per topic} -- drove the \texttt{user\_progress} table and the progress page with the 30-day activity heatmap. Visual progress indicators have been shown to increase learner motivation and persistence \cite{hattie2007}.
  \item \textbf{Admin dashboard for user and content management} -- drove the admin user management page and the safeguard that prevents deletion of admin accounts.
\end{itemize}

These design decisions are described in full in the System Architecture (Chapter 4) and Database Design (Chapter 5) chapters.

\section{User Requirements}
\label{sec:user-req}

The system has two distinct user roles: \textit{Learner} (a regular registered user) and \textit{Administrator}. The requirements below reflect the goals and expectations of each role, gathered through informal interviews and self-directed testing during development.

\subsection{Learner (Regular User)}

The learner's primary interaction with the platform revolves around discovering, consuming, and rating learning resources. The key needs expressed during requirement gathering were:

\begin{itemize}
  \item I want to see a clear path of what topics to study and in what order.
  \item I want to rate resources so I can remember which ones were useful.
  \item I want the platform to suggest resources I have not seen yet, based on what I have done.
  \item I want to track my progress and see how much of a course I have completed.
  \item I want to contribute resources I found useful so other learners can benefit.
\end{itemize}

These needs map directly to FR-04, FR-07, FR-11, FR-09, and FR-06 respectively.

\subsection{Administrator}

The administrator's role is to maintain the quality of the content library and the integrity of the user base. The requirements identified for this role were:

\begin{itemize}
  \item I want to review submitted resources before they go live to maintain quality.
  \item I want to see all registered users and be able to remove bad actors.
  \item I want the platform to prevent me from accidentally deleting admin accounts.
\end{itemize}

These needs map to FR-12, FR-13, and the guard condition in the admin user management endpoint.

\chapter{Methodology}

This chapter describes the development approach, version control strategy, and testing methodology used throughout the project. The choices made here had a direct impact on how quickly integration issues were identified and resolved during the ten-week build.

\section{Development Approach}

This project was developed over approximately three months, from early April to late June 2026. Rather than following a strict waterfall model, development was organised into an \textbf{iterative, feature-driven} process. Work progressed in phases, each centred on a specific feature group, so that each feature was working end-to-end before the next one began. Integration issues were therefore caught early rather than accumulating into a large unresolved backlog at the end.

Table~\ref{tab:timeline} summarises the six phases of development and the work completed in each.

The iterative approach also meant that the database schema was informed by real usage patterns observed during development, rather than being designed entirely upfront. The \texttt{version} column on the \texttt{resources} table, for example, was added during the May review phase when it became clear that resource update support would be needed in a future iteration.

\section{Version Control}

Git was used throughout the project with commits grouped by feature. The repository is hosted on GitHub, which also serves as the deployment trigger -- both the backend (Render) and frontend (Vercel) automatically redeploy on every push to the \texttt{main} branch. This created a simple but effective CI/CD pipeline. A GitHub Actions workflow additionally runs the automated test suite on every push, preventing regressions from being merged undetected.

\section{Testing Strategy}

Two layers of testing were applied to verify correctness:

\begin{enumerate}
  \item \textbf{Automated unit tests:} Written using \texttt{pytest} with \texttt{httpx} for the FastAPI test client. Tests cover authentication flows (registration, login, token validation) and the recommendation algorithm (correct scoring formula, cold-start fallback behaviour).
  \item \textbf{Manual end-to-end testing:} The deployed system was tested by multiple users across different devices and browsers. Reported issues -- including the CORS middleware ordering bug and the missing React Router entry for \texttt{/topics} -- were resolved before the final submission. These incidents are documented in detail in Chapter 9.
\end{enumerate}

\chapter{System Architecture}

This chapter describes how the Opic system is structured at a high level, how each layer communicates, and where each component is deployed. The architecture was chosen to satisfy NFR-06 (maintainability through separation of concerns) and to allow the frontend and backend to be deployed, scaled, and updated independently.

\section{High-Level Architecture}

Opic follows a classic \textbf{client-server} architecture with a clear separation between the frontend and backend. The frontend is a browser-based single-page application; the backend is a REST API server; and the database is a managed relational store. Figure~\ref{fig:arch} illustrates how these three layers interact.

\begin{figure}[H]
\centering
\IfFileExists{arch.png}{\includegraphics[width=0.85\textwidth]{arch.png}}{\missingfig{arch.png}{System Architecture Diagram -- showing React SPA communicating with FastAPI over HTTPS, which in turn reads from and writes to PostgreSQL}}
\caption{System Architecture Diagram}
\label{fig:arch}
\end{figure}

All communication between the frontend and the backend occurs over HTTPS using JSON-encoded REST requests. There is no WebSocket or real-time connection; all state is fetched on demand. This keeps the architecture simple and makes the API straightforwardly testable.

\section{Frontend}

The frontend is a \textbf{React} single-page application (SPA) built with \textbf{Vite}. It communicates with the backend exclusively through REST API calls using \texttt{axios}. React Router v6 handles client-side navigation so that page transitions do not trigger full reloads.

Authentication state (the logged-in user's token and profile) is managed through a React Context provider, making it available to all components without prop drilling. The context is initialised from \texttt{localStorage} on page load, so the session survives browser refreshes.

\section{Backend}

The backend is built with \textbf{FastAPI}, a modern Python web framework that generates interactive OpenAPI documentation automatically at \texttt{/docs}. The application is structured into five layers:

\begin{itemize}
  \item \texttt{app/api/} -- Route handlers for each feature group (\texttt{auth}, \texttt{topics}, \texttt{resources}, \texttt{recommend}).
  \item \texttt{app/models/} -- SQLAlchemy ORM models, one file per database table.
  \item \texttt{app/schemas/} -- Pydantic request and response schemas for input validation and serialisation.
  \item \texttt{app/services/} -- Business logic, most importantly the recommender service.
  \item \texttt{app/core/} -- Database connection setup and security utilities (JWT signing, password hashing).
\end{itemize}

This layered structure ensures that route handlers remain thin and that business logic is reusable and independently testable.

\section{Deployment Architecture}

The system is deployed entirely on free-tier cloud infrastructure. Table~\ref{tab:deploy} summarises each component and the service it runs on.

\begin{longtable}{p{0.2\textwidth} p{0.3\textwidth} p{0.4\textwidth}}
\caption{Deployment Infrastructure}\label{tab:deploy}\\
\toprule
\textbf{Component} & \textbf{Service} & \textbf{Notes} \\
\midrule
Frontend & Vercel & Auto-deploys on git push. Static bundle served from CDN. \\
Backend & Render & Python web service. Persistent process. \\
Database & Render PostgreSQL & Managed PostgreSQL. Automatic backups. \\

\end{longtable}

The frontend is built as a static bundle and served from Vercel's global CDN. The backend runs as a persistent web service on Render, with CORS configured to only accept requests from the known Vercel domains. This ensures that cross-origin requests from any other source are rejected at the HTTP level. The deployment pipeline is fully automated: a push to the \texttt{main} branch triggers a GitHub Actions CI run, and if tests pass, both Vercel and Render pick up the new commit and redeploy automatically.

\section{Key Design Decisions}

Several non-obvious choices were made during schema design. The rationale for the most important ones is summarised below:

\begin{itemize}
  \item \textbf{Composite primary keys} on \texttt{ratings} and \texttt{user\_progress} prevent duplicate rows naturally without requiring application-level uniqueness checks.
  \item \textbf{Indexed columns:} \texttt{(topic\_id, status)} on resources and \texttt{(user\_id, completed)} on engagements speed up the most frequent queries (fetching approved resources for a topic, and building the recommendation matrix).
  \item \textbf{Soft/hard prerequisites:} The \texttt{is\_hard} boolean distinguishes mandatory prerequisites from suggested ones, allowing the UI to differentiate them in a future iteration.
  \item \textbf{Resource versioning:} The \texttt{version} column was included for future support of resource updates without losing rating history. It is not yet used by the current implementation.
  \item \textbf{Rating computed at query time:} Rather than storing a pre-computed average, the overall rating is calculated from the \texttt{ratings} table on each request. This ensures the displayed value is always accurate immediately after any new rating is submitted, without requiring a background synchronisation job.
\end{itemize}

\begin{equation}
  \text{score} = 0.35 \cdot \frac{\text{stars}}{5} + 0.25 \cdot w_{\text{watch}} + 0.20 \cdot \frac{\min(t, 600)}{600} + 0.10 \cdot r_{\text{revisit}} + 0.10 \cdot c_{\text{complete}}
\end{equation}

where $w_{\text{watch}}$ is the watch completion ratio (0--1), $t$ is time spent in seconds capped at 600, $r_{\text{revisit}}$ is a normalised revisit count, and $c_{\text{complete}}$ is 1 if the resource was marked complete. The weights were set to reflect the relative reliability of each signal: explicit star ratings carry the most weight (35\%), while completion and revisit signals serve as supporting evidence.

Cosine similarity is then computed between the target user's row vector and every other user's row:

\begin{equation}
  \text{sim}(u, v) = \frac{\mathbf{u} \cdot \mathbf{v}}{\|\mathbf{u}\| \cdot \|\mathbf{v}\|}
\end{equation}

The predicted score for a candidate resource $r$ not yet seen by user $u$ is a weighted combination of the collaborative score and a popularity baseline:

\begin{equation}
  \hat{s}(u, r) = 0.6 \cdot \frac{\sum_{v \neq u} \text{sim}(u,v) \cdot s(v,r)}{\sum_{v \neq u} |\text{sim}(u,v)|} + 0.4 \cdot p_r
\end{equation}

where $p_r$ is the Bayesian average popularity score of the resource. The 60/40 split between collaborative and popularity scores prevents the engine from over-fitting to a small number of similar users.

\subsection{Bayesian Average for Popularity}

To avoid the cold-resource problem (new resources with very few ratings appearing unrealistically high or low in rankings), a Bayesian average is used as the popularity baseline:

\begin{equation}
  \bar{r}_{\text{Bayes}} = \frac{m \cdot C + \sum_{i} r_i}{m + n}
\end{equation}

where $m = 5$ is the prior count, $C = 3.0$ is the prior mean rating, $n$ is the number of actual ratings, and $r_i$ are the individual star ratings. With these values, a resource with only one 5-star rating receives a Bayesian average of $\frac{5 \times 3 + 5}{5 + 1} \approx 3.33$ rather than 5.0, preventing it from dominating the ranking until it has collected sufficient evidence.

\subsection{Cold-Start Fallback}

When a new user has no ratings, cosine similarity cannot be computed. The system falls back to the Bayesian popularity ranking and boosts resources whose topics match the user's onboarding quiz answers:

\begin{lstlisting}[language=Python, caption=Cold-start topic boosting]
if resource.topic_id in boosted_topics:
    base_score += 1.5
\end{lstlisting}

Topics are only boosted if the user gave a positive answer to the corresponding onboarding question. Answers containing negative words (\texttt{no}, \texttt{never}, \texttt{don't}) are excluded from the boost set to avoid recommending content the user explicitly said they are not interested in.

\subsection{Gemini Integration}

The backend sends a structured prompt to the Gemini API requesting five topic-specific multiple choice questions with plausible distractors and varied correct-answer positions. If the Gemini API is unavailable or returns malformed JSON, the endpoint falls back to locally-generated questions derived from the topic title and description. This ensures the quiz always appears and never blocks topic completion, satisfying FR-15.

\section{Admin Review Workflow}
\label{sec:impl-adminreview}

Resources submitted by users enter a \texttt{pending} state and are invisible to other learners. The admin review page (UC-08) lists all pending submissions, showing each resource's title, URL, submitter, and associated topic. An admin can approve or reject each submission by clicking the corresponding button. Only approved resources appear in topic pages.

The admin user management page (UC-09) lists all registered accounts. Admins can delete any non-admin account. A guard condition in the delete endpoint prevents accidental removal of admin accounts -- attempting to delete an admin account returns HTTP 403 with a descriptive error message.

\section{System Evaluation}
\label{sec:results-eval}

\subsection{Recommendation Quality}

To verify the recommendation engine, two test accounts were created with deliberately different interaction histories. Each account rated five resources in different topic areas. After these interactions, the two accounts received noticeably different sets of recommendations, confirming that the collaborative filtering component correctly differentiates users based on their behaviour.

For a brand-new account with no ratings, the system correctly fell back to popularity-based ranking boosted by onboarding quiz answers, confirming that the cold-start handling works as designed.

The Bayesian average formula also behaved correctly: resources with only one or two ratings did not rank unrealistically high or low, as the prior mean of 3.0 pulled extreme values toward the centre until sufficient ratings were collected.

\subsection{Non-Functional Requirements}

The non-functional requirements from Chapter 2 were evaluated as follows:

\begin{itemize}
  \item \textbf{NFR-01/NFR-02 (Security):} Verified by inspecting the database -- passwords are stored as bcrypt hashes. Unauthenticated requests to protected endpoints return HTTP 401. Requests to admin endpoints from non-admin users return HTTP 403.
  \item \textbf{NFR-03 (Performance):} API response times for non-recommendation endpoints were measured at under 200\,ms for typical payloads on the Render free tier, meeting the 300\,ms target.
  \item \textbf{NFR-04 (Usability):} The platform was tested on desktop Chrome, mobile Safari, and mobile Chrome. All core flows were functional on all three.
  \item \textbf{NFR-06 (Maintainability):} The frontend was updated and redeployed independently of the backend on multiple occasions during the project, confirming the separation of deployable units.
  \item \textbf{NFR-08 (CORS):} Requests from origins not in the Vercel allowlist are rejected at the API level. This was verified by sending test requests from a localhost origin not in the allowlist.
\end{itemize}

\section{Bugs Discovered and Resolved}
\label{sec:results-bugs}

Two notable bugs were discovered during the evaluation phase and resolved before the final submission:

\begin{itemize}
  \item \textbf{CORS preflight failure in production:} FastAPI processes middleware in reverse registration order. The custom logging middleware was registered after \texttt{CORSMiddleware}, causing it to execute first and return error responses before CORS headers could be attached. This was fixed by reordering middleware registration so that \texttt{CORSMiddleware} runs outermost. The bug only appeared in the deployed environment because the development server does not enforce the same CORS origin restrictions.
  \item \textbf{Missing \texttt{/topics} route:} After onboarding quiz completion, the frontend redirected to \texttt{/topics}, but this route was never registered in the React Router configuration. The \texttt{Topics} component existed but was disconnected. This was caught during user testing and fixed by adding the route to \texttt{App.jsx}.
\end{itemize}

Both bugs were caught through manual end-to-end testing on the live deployment, highlighting the value of testing the deployed system rather than relying solely on automated tests in a local environment.

\chapter{Challenges and Limitations}

This chapter reflects honestly on the difficulties encountered during development and the known limitations of the current implementation. Understanding these constraints is important for anyone who wishes to extend or maintain the system.

\section{Challenges Encountered}

\textbf{CORS middleware ordering.} One of the more frustrating bugs during deployment was a CORS error that only appeared in production. As described in Chapter 8, the root cause was that FastAPI processes middleware in reverse registration order. The fix -- reordering the middleware registration -- took less than a minute to implement once the cause was understood; the difficulty lay entirely in diagnosing it without helpful error messages.

\textbf{Missing route.} After deployment, the onboarding quiz redirected to \texttt{/topics}, but this route was never added to the React Router configuration. The \texttt{Topics} component existed but was not connected. This was a simple oversight caught during user testing.

\textbf{Cold-start problem.} New users with no ratings cannot benefit from collaborative filtering. The onboarding quiz partially addresses this by providing topic-level preference signals, but the quality of cold-start recommendations is still lower than for active users. This is a known and accepted limitation of user-based collaborative filtering systems.

\textbf{Render free tier spin-down.} The Render free tier spins down the backend after 15 minutes of inactivity. The first request after a period of no activity can take 30--60 seconds while the service wakes up. This is not ideal for a live demonstration and would need a paid tier to resolve in a production setting.

\section{Limitations}

The current implementation has several known limitations that were intentionally deferred to future work:

\begin{itemize}
  \item \textbf{No email notifications for review decisions:} When a resource is approved or rejected, the submitter is not automatically notified. Users must check the ``My Submissions'' section of their profile manually. The SMTP infrastructure for sending emails already exists in the codebase (used for password reset) and could be reused with minimal effort.
  \item \textbf{Scalability of the recommendation engine:} The recommender builds the full user-resource matrix on every API request. For the current user base this is acceptable, but it would not scale to thousands of users without pre-computation or caching.
  \item \textbf{No content-based filtering:} Recommendations are based entirely on user behaviour. Two resources with identical titles but different quality would be treated identically by the engine. Adding content-based features (topic tags, resource type, title keywords) would improve recommendations for newly added resources.
  \item \textbf{Single binary admin role:} The \texttt{is\_admin} flag is a boolean. There is no moderator-level role with partial permissions, making it impossible to delegate content review without granting full admin access.
\end{itemize}

These limitations represent natural starting points for future work, as described in the following chapter.

\chapter{Conclusion and Future Work}

\section{Conclusion}

Opic was designed to solve a real and familiar problem: the difficulty of navigating a vast sea of online learning material as a student. The platform was successfully built and deployed within the planned timeframe, delivering all fourteen intended functional requirements.

The most technically interesting component is the recommendation engine, which combines user-based collaborative filtering with a Bayesian average popularity score and a cold-start fallback driven by onboarding quiz answers. The result is a system that provides useful recommendations even for new users and improves meaningfully as users interact with more content.

The project also demonstrated the value of a clean architecture. Separating the frontend, backend, and database into independent deployable units made it straightforward to host the entire system on free-tier cloud services without any infrastructure cost, and allowed each layer to be updated and redeployed independently throughout the development cycle.

\section{Future Work}

Given more time, the following improvements would be the most impactful:

\begin{enumerate}
  \item \textbf{JWT refresh tokens:} The current implementation issues a single access token that expires after a fixed period, silently logging out the user. A refresh token flow would maintain sessions more gracefully without compromising security.

  \item \textbf{Email notifications for review decisions:} Notify users automatically when their resource submission is approved or rejected. The SMTP infrastructure already exists in the codebase and could be reused with minimal effort.

  \item \textbf{Recommendation caching:} The recommendation engine currently rebuilds the full user-resource matrix on every API call. Pre-computing and caching the matrix on a fixed schedule (e.g.\ every 30 minutes) would significantly reduce response times at scale.

  \item \textbf{Hybrid content-based filtering:} Adding content-based features -- resource type, topic tags, title keywords -- would improve recommendations for new resources that have not yet been rated by many users.

  \item \textbf{Pagination:} Several endpoints return all records in a single response. Adding limit/offset pagination would be necessary for a production-scale content library.

  \item \textbf{Mobile application:} A React Native version of the frontend would provide a better experience for users who prefer to learn on their phones.

  \item \textbf{Analytics dashboard:} An admin view showing platform-wide statistics -- most popular topics, average completion rate, rating distributions, and daily active users -- would help administrators make data-driven content decisions.
\end{enumerate}

\chapter*{Abstract}
\addcontentsline{toc}{chapter}{Abstract}

Finding good learning material online is easy. Knowing what to study first, what order makes sense, and which results are actually useful -- that is the harder problem. Opic was built to address this for university students, particularly in ICT, Data Science, and BIT programmes at USTH.

The platform organises content into topic roadmaps with prerequisite links, lets users submit and rate resources, and ranks materials by relevance to each individual. Technically, it is a React frontend talking to a FastAPI backend over REST, with a PostgreSQL database. The ranking uses user-based collaborative filtering with cosine similarity and a Bayesian average to handle sparse ratings. New users are handled through an onboarding quiz that seeds the cold-start fallback. When marking a topic complete, users take a short Gemini-generated quiz; three or more wrong answers sends them back to review first. Completing all topics in a course unlocks a longer final exam (up to 10 questions, including coding challenges for programming courses) that must be passed at 50\% or above before the course is marked complete.

The system is deployed on Render (backend) and Vercel (frontend). All sixteen functional requirements are working on the live deployment.

\textbf{Keywords:} personalised ranking, collaborative filtering, learning platform, FastAPI, React, cold-start, Bayesian average, knowledge quiz

\chapter{Introduction}

\section{Context and Motivation}

There is no shortage of learning material online. YouTube, Coursera, and Medium alone publish thousands of tutorials and articles every week \cite{coursera2023}. For a student trying to pick up a new topic, the real problem is not finding content -- it is figuring out what to study first, what order makes sense, and which results are actually worth the time.

Drachsler and Greller \cite{drachsler2016} call this the ``paradox of choice'': more resources does not mean better learning, and too many options can increase cognitive load and leave learners more confused than when they started. Burke \cite{burke2002} made a similar point -- web-based learning environments have long struggled to show people content relevant to them specifically, rather than just popular in general.

Recommender systems have addressed this since the mid-1990s, starting with the GroupLens project \cite{resnick1994}. Collaborative filtering is now a standard approach for personalised ranking \cite{koren2009}, used at scale by platforms like Netflix and Spotify. Opic applies a simplified version of these ideas -- not at that scale, but enough to rank resources differently for different users based on their rating behaviour.

Opic was built for university students, particularly in ICT, Data Science, and BIT programmes at USTH, where students are expected to do substantial self-directed study \cite{zimmerman2002,brusilovsky2007}. The name comes from \textit{topic}, the core unit of learning in the system.

\section{Objectives}

This project set out to address three specific problems:

\begin{enumerate}
  \item \textbf{Navigation} -- not knowing what to study or in what order.
  \item \textbf{Quality} -- not knowing which resources are actually good.
  \item \textbf{Personalisation} -- getting the same results as everyone else regardless of background.
\end{enumerate}

These are addressed through topic roadmaps with prerequisite links, a community submission and admin review pipeline, and user-based collaborative filtering for ranking. CF was chosen over content-based filtering (which needs expert tagging) and matrix factorisation (which requires offline training) because it is simpler, interpretable, and works directly on live rating data \cite{herlocker2004}. The cold-start problem for new users is partially mitigated via an onboarding quiz \cite{schein2002}.

\section{Expected Outcome}

By the end of the project the following were working and deployed:

\begin{itemize}
  \item A live web application where users can register, take the onboarding quiz, browse topic roadmaps, and access curated resources.
  \item Personalised ranking that produces different results for different users and works from day one via the cold-start fallback.
  \item An admin content pipeline: users submit resources, admins approve or reject them, only approved resources appear on topic pages.
  \item Topic-level progress tracking with a 30-day activity heatmap.
  \item JWT authentication with email-based password reset \cite{ietfjwt}. Passwords stored as bcrypt hashes per OWASP guidelines \cite{owasp}.
  \item Deployment on Vercel (frontend) and Render (backend + database) with automatic redeploy on every git push.
\end{itemize}

Chapter~2 covers requirements and use cases. Chapter~3 describes the architecture, database design, implementation details, and the tools used. Chapter~4 presents results, evaluation, and discussion including challenges and limitations. Chapter~5 concludes.

Instead of designing everything upfront and building at the end, the work was split into feature phases \cite{beck2001} -- each phase had one clear goal: get a group of features working end-to-end before moving on. This meant bugs and integration problems surfaced early, when they were still easy to fix. Table~\ref{tab:timeline} shows the breakdown.

Git was used throughout with commits grouped by feature. Both Render and Vercel auto-deploy on every push to \texttt{main}, with a GitHub Actions workflow running the test suite first so a failing test blocks the deploy. Testing was done at two levels: automated \texttt{pytest} tests covering the auth flows and ranking algorithm, and manual end-to-end testing on the live site by classmates across different devices and browsers.

\section{System Architecture}

Opic is a standard client-server application. The frontend is a React SPA, the backend is a FastAPI REST API, and the database is PostgreSQL. Figure~\ref{fig:arch} shows how these three layers connect. All communication is over HTTPS with JSON -- there is no WebSocket, everything is request/response. This keeps the system simple and easy to test.

\begin{figure}[H]
\centering
\IfFileExists{arch.png}{\includegraphics[width=0.82\textwidth]{arch.png}}{\missingfig{arch.png}{System Architecture Diagram}}
\caption{System Architecture Diagram}
\label{fig:arch}
\end{figure}

The frontend is built with React 18 and Vite \cite{vite}. \texttt{axios} handles all API calls with an interceptor that attaches the JWT token to every request. React Router v6 handles navigation. Auth state is stored in a React Context, initialised from \texttt{localStorage} on page load so sessions survive refreshes.

The backend uses FastAPI \cite{fastapi} split into four layers: \texttt{app/api/} (route handlers), \texttt{app/models/} (SQLAlchemy ORM \cite{sqlalchemy}), \texttt{app/services/} (ranking logic), and \texttt{app/core/} (database setup, JWT, password hashing). FastAPI generates interactive API docs automatically at \texttt{/docs}, which was useful during development.

Deployment runs on free-tier cloud services: Vercel hosts the frontend as a static bundle on a global CDN; Render hosts the backend as a persistent Python web service alongside a managed PostgreSQL instance. Both redeploy automatically on every push to \texttt{main} \cite{githubactions}.

\section{Use Case Implementation}

\subsection{Authentication}
\label{sec:impl-auth}

Authentication covers three use cases: registration (UC-02), login (UC-01), and password reset (UC-03). The onboarding quiz (UC-06, FR-10) runs immediately after registration to seed the cold-start recommendation fallback. Together these define the security boundary for the entire system.

\begin{equation}
  \label{eq:score}
  \text{score} = 0.35 \cdot \frac{\text{stars}}{5} + 0.25 \cdot w_{\text{watch}} + 0.20 \cdot \frac{\min(t, 600)}{600} + 0.10 \cdot r_{\text{revisit}} + 0.10 \cdot c_{\text{complete}}
\end{equation}

Cosine similarity is then computed between each pair of user row vectors:
\begin{equation}
  \text{sim}(u, v) = \frac{\mathbf{u} \cdot \mathbf{v}}{\|\mathbf{u}\| \cdot \|\mathbf{v}\|}
\end{equation}

The predicted score for a candidate resource $r$ not yet seen by user $u$ blends the collaborative signal with a Bayesian popularity baseline:
\begin{equation}
  \hat{s}(u, r) = 0.6 \cdot \frac{\sum_{v \neq u} \text{sim}(u,v) \cdot s(v,r)}{\sum_{v \neq u} |\text{sim}(u,v)|} + 0.4 \cdot p_r
\end{equation}

The 60/40 split prevents over-fitting to a small number of similar users. The Bayesian average $p_r$ handles new resources with few ratings:
\begin{equation}
  \bar{r}_{\text{Bayes}} = \frac{m \cdot C + \sum_{i} r_i}{m + n}
\end{equation}

with $m = 5$ (prior count) and $C = 3.0$ (prior mean). A resource with one 5-star rating receives $\approx 3.33$ rather than 5.0, preventing it from dominating the ranking until it has enough evidence.

\textbf{Cold-start fallback.} When a new user has no ratings, cosine similarity cannot be computed. The system falls back to Bayesian popularity ranking and boosts resources whose topics match the user's onboarding quiz answers:

\begin{lstlisting}[language=Python, caption=Cold-start topic boosting]
if resource.topic_id in boosted_topics:
    base_score += 1.5
\end{lstlisting}

Topics are only boosted for positive answers. Answers containing negative words (\texttt{no}, \texttt{never}, \texttt{don't}) are excluded to avoid recommending content the user explicitly said they are not interested in.

\subsection{AI Knowledge Quiz}
\label{sec:impl-quiz}

When a user clicks ``Mark Complete'' on a topic, the backend calls the Google Gemini 2.0 Flash API \cite{gemini2024} to generate five multiple choice questions tailored to the topic's title and description. The quiz is shown in a modal before the topic is recorded as complete in the database.

Figure~\ref{fig:quiz-flow} shows the complete decision flow from clicking ``Mark Complete'' to topic completion. The quiz presents one question at a time. The user selects one of four options and clicks Confirm to reveal the result -- correct answers highlight green, wrong answers highlight red with the correct answer also shown. A running wrong-answer counter is visible in the modal header. If the user accumulates three or more incorrect answers the quiz ends with a retry screen offering either ``Try Again'' or ``Go Back'' to review the material first. If the user passes, a score summary is shown and a ``Mark as Complete'' button finalises the topic completion. The quiz cannot be bypassed -- there is no skip option.

\section{System Evaluation}
\label{sec:results-eval}

\subsection{Recommendation Quality}

To verify the ranking feature, two test accounts were created with deliberately different rating histories -- each rating five resources in different topic areas. After these interactions, the two accounts received noticeably different suggestion sets, confirming that the collaborative filtering logic differentiates users based on their behaviour.

A brand-new account with no ratings correctly fell back to popularity-based ranking boosted by onboarding answers, confirming the cold-start handling works as designed. The Bayesian average formula also behaved correctly: resources with only one or two ratings did not rank unrealistically high or low, as the prior mean of 3.0 pulled extreme values toward centre until sufficient ratings were collected.

\subsection{Non-Functional Requirements}

\begin{itemize}
  \item \textbf{NFR-01/NFR-02 (Security):} Passwords confirmed stored as bcrypt hashes in the database. Unauthenticated requests to protected endpoints return HTTP 401. Non-admin requests to admin endpoints return HTTP 403.
  \item \textbf{NFR-03 (Performance):} Non-recommendation API responses measured at under 200\,ms on the Render free tier, meeting the 300\,ms target.
  \item \textbf{NFR-04 (Usability):} Tested on desktop Chrome, mobile Safari, and mobile Chrome -- all core flows worked correctly.
  \item \textbf{NFR-06 (Maintainability):} Frontend was updated and redeployed independently of the backend multiple times during development, confirming the separation of deployable units.
  \item \textbf{NFR-08 (CORS):} Requests from origins not in the allowlist are rejected at the API level, verified by sending test requests from a non-listed localhost origin.
\end{itemize}

\section{Bugs Discovered and Resolved}
\label{sec:results-bugs}

Two notable bugs were discovered during the evaluation phase and resolved before final submission:

\begin{itemize}
  \item \textbf{CORS preflight failure in production.} FastAPI processes middleware in reverse registration order. The custom logging middleware was registered after \texttt{CORSMiddleware}, causing it to execute first and return error responses before CORS headers could be attached. This produced CORS errors in the browser on the deployed site, but not locally, because the development server does not enforce the same origin restrictions. Fixed by reordering middleware registration so \texttt{CORSMiddleware} runs outermost.
  \item \textbf{Missing \texttt{/topics} route.} After onboarding quiz completion, the frontend redirected to \texttt{/topics}, but that route was never registered in the React Router configuration. The \texttt{Topics} component existed but was not connected to any route. Caught during user testing and fixed by adding the route in \texttt{App.jsx}.
\end{itemize}

Both bugs were caught through manual end-to-end testing on the live deployment, highlighting the value of testing the deployed system rather than relying solely on automated tests in a local environment.

\section{Challenges and Limitations}
\label{sec:challenges}

\subsection{Challenges}

\textbf{CORS middleware ordering.} The CORS error appearing in production but not locally took time to track down. Nothing in the error message pointed to middleware ordering -- it was found only by reading the FastAPI documentation carefully. The frustrating part was that the fix was a one-line reorder.

\textbf{Cold-start quality.} New users with no ratings get popularity-based recommendations boosted by their onboarding answers. This works, but the suggestions are noticeably less relevant than what active users see. The quiz helps but does not fully close the gap -- the engine has essentially no personalisation data to work with until the user rates a few resources.

\textbf{Render free tier spin-down.} Render shuts down the backend after 15 minutes of no traffic. The first request after that takes 30--60 seconds to wake up. This is fine for normal use, but awkward during a live demo.

\subsection{Limitations}

\begin{itemize}
  \item \textbf{No submission notifications.} Users must check their profile page to see if a resource was approved or rejected. The SMTP infrastructure from the password reset flow could handle this easily -- it just was not prioritised.
  \item \textbf{Recommender scalability.} The full user-resource matrix is rebuilt on every request. This works at the current user count, but would not scale to thousands of users. Pre-computing the matrix every 30 minutes would help significantly.
  \item \textbf{No content-based features.} The engine only looks at user behaviour. A new resource with no ratings is treated the same as any other unrated resource regardless of how relevant it is to the topic.
  \item \textbf{Binary admin role.} Admin is a simple boolean flag. There is no way to grant review permissions without making a user a full admin with access to all admin functions.
\end{itemize}

\chapter{Conclusion and Future Work}

\section{Conclusion}

Opic was built in response to a straightforward problem: as a student studying ICT, it is genuinely hard to figure out what to learn next and whether a given tutorial is worth the time. The platform was built and deployed within the planned three months and met all fifteen functional requirements.

The most interesting part to build was the ranking feature. Getting anything useful from a sparse rating matrix -- and having it still work for new users with no history -- required combining collaborative filtering with a Bayesian popularity score and an onboarding quiz fallback. It is a simple implementation by industry standards, but it produces noticeably different suggestions for different users, which was the goal.

The decision to keep the frontend and backend completely separate also paid off. Being able to push UI changes to Vercel without touching the Render backend made the final weeks of bug fixing much more manageable.

\section{Future Work}

\begin{enumerate}
  \item \textbf{JWT refresh tokens.} Users are silently logged out when their token expires. A refresh token would fix this without weakening security.
  \item \textbf{Submission notifications.} Notify submitters by email when a resource is approved or rejected. The SMTP infrastructure from the password reset flow is already in place.
  \item \textbf{Cache the recommendation matrix.} Pre-compute the user-resource matrix every 30 minutes rather than rebuilding on every request. This is the most important change for scalability.
  \item \textbf{Content-based features.} Adding resource type, topic tags, or title keywords would improve recommendations for new resources that have not yet been rated.
  \item \textbf{Pagination.} Several endpoints return every record in one response -- necessary for a real deployment.
  \item \textbf{Admin analytics.} A dashboard showing popular topics, average completion rates, and daily active users would help admins prioritise content.
\end{enumerate}
